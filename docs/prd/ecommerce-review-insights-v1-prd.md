## 一、产品概述

### 1.1 产品定位

面向电商运营的 B/S 架构分析系统，通过**配置的评论抓取 API** 获取商品评论，进行**情感分析**与**六维痛点分析**，为选品、优化卖点、客服与退货归因提供数据支撑，并支持多商品对比与可视化决策。分析能力由**接口配置**选用**自建/微调模型部署**或**外部大模型 API**（统一通过「分析用模型/大模型 API」配置项对接，用于**评论洞察**）；**智能 Agent API** 仅用于**对比分析**中的结论卡片。**界面语言**暂时支持用户选择 **English** 或**简体中文**（菜单、按钮、系统文案）；**分析与展示**相关文本统一以英文输出（模型的训练主要是通过英文）；当界面语言非 English 时，将英文分析文本翻译为对应界面语言并以上下结构展示；**证据句**区域仅展示对应原始评论片段（不翻译）。

### 1.2 核心价值

- **评论洞察**：输入平台商品标识（`platform + product_id`）或商品网址 → 接入评论抓取 API 获取评论 → 情感 + 六维痛点分析 → 存储与可视化（六维下的痛点排行、维度词云、证据句高亮、时间趋势、筛选与导出）；分析结果相关文本统一以英文展示；当界面语言非 English 时，在英文下方紧接展示译文（上下结构，机器翻译提示）；证据句区域仅展示对应原始评论片段（不翻译）（见 4.5）。
- **对比分析**：双商品差异化对比（情感、痛点、关键词、结论卡片），辅助选品与竞品分析；分析结果相关文本统一以英文展示；当界面语言非 English 时，在英文下方紧接展示译文（上下结构，机器翻译提示）；若对比页支持证据句下钻，则证据句区域仅展示对应原始评论片段（不翻译）。
- **接口配置**：评论抓取 API 必配；**分析用模型/大模型 API**（评论洞察用）可配置为自建/微调模型部署或对接外部大模型；**智能 Agent API**（对比分析结论用）可选；**翻译 API**（按界面语言目标语种）；当界面语言非 English 且未配置翻译 API 时仅展示英文并提示「翻译服务未配置」；详见 2.5、4.3 与 4.5。
- **系统设置**：系统设置支持用户与权限管理，满足团队使用。

### 1.3 目标用户

- **电商运营 / 选品人员**：看评论洞察与对比，做选品与卖点优化。
- **产品/客服负责人**：看退货原因、用户期望、应用场景，改进产品与话术。
- **系统管理员**：配置接口、管理账号与权限。

### 1.4 产品边界与假设

本节明确本产品「做哪些、不做哪些」以及依赖的前提条件，用于范围控制和与后续 spec/plan 的衔接。

**在范围内（本产品负责实现）**

- **数据接入**：通过配置并调用外部评论抓取 API 获取评论数据并落库；不实现爬虫与反爬逻辑本身。
- **分析能力**：情感分析（准确率≥85%）与六维痛点分析（优点、缺点、退货原因、购买动机、用户期望、应用场景）；支持自建/微调模型与外部大模型 API 并存：管理员在接口配置维护多个分析源，业务用户在「评论洞察」输入区底部按任务选择本次分析源。
- **结果呈现**：痛点评分与排序、证据句关联与高亮、六维下的痛点排行与维度词云、时间趋势（评论带时间戳，按年/月/日展示）、筛选与导出；带 SKU 则支持按 SKU 看六维分布；**全站界面语言**可选 English / 简体中文；**分析与可视化文本**统一以英文展示；当界面语言非 English 时，在英文下方紧接展示译文（上下结构）；**证据句**区域仅展示对应原始评论片段（不翻译）。
- **对比与增强**：双商品对比（情感、痛点、关键词、结论卡片）；可选配置智能 Agent API 用于结论生成等增强。
- **配置与权限**：评论抓取 API、**分析用模型/大模型 API**、智能 Agent API、**翻译 API**的配置管理；用户与权限管理（账号、角色、功能级权限）。

**不在本版范围（明确不做）**

- **评论抓取端**：爬虫开发、反爬与反反爬、自建抓取集群或调度；评论数据仅通过「配置好的外部 API」获取。
- **标注与训练平台**：独立的标注工具/平台、模型训练/实验平台的产品化；训练与标注在系统外完成，本系统负责「使用」模型或调用已配置的 API。
- **其他**：不承诺实现评论抓取 API 的提供方逻辑、不负责第三方 API 的可用性与合规。

**假设（后续设计与实现可依赖的前提）**

- **数据**：模型训练/微调采用**直接下载的标注数据集**（公开或自有），与评论抓取 API 的实时数据分离——API 仅用于生产环境按商品标识（`platform + product_id`）获取评论，训练与评估不依赖 API 实时抓取。评论抓取 API 的请求协议与返回字段可通过接口配置约定并与系统对齐。
- **平台与扩展**：首版即按**多站点/多语种全局化架构**设计与实现（站点通过配置扩展，不限定单一国家站点）。对非英文站点：分析阶段先将站点文本翻译为英文作为模型输入；展示阶段仍以英文分析结果为主，并在界面语言非 English 时展示英文译文；证据句区域始终展示原始评论（不翻译）。
- **模型与配置**：分析能力可通过配置切换「自建/微调模型」与「外部大模型 API」，无需为每种来源单独定制一套产品逻辑。

### 1.5 实施顺序

项目实施按以下阶段推进（阶段目标级描述）：

- **阶段一：模型能力就绪**——完成情感与六维痛点分析能力并达到既定结果指标（如情感准确率≥85%）；新痛点发现与词典回灌可作为可选离线能力纳入后续迭代。
- **阶段二：产品化落地**——在模型能力可用后，完成前后端与数据链路的产品化，覆盖评论洞察、对比分析、接口配置、系统设置等核心模块并支撑上线运行。

### 1.6 MVP 范围与目标（已澄清）

- **MVP 必须上线能力**：评论洞察、对比分析、多分析源任务级选择、翻译展示、完整 RBAC（基于预置角色模板）。
- **业务优先目标（Why）**：优先提升**问题定位准确度（痛点归因）**，以证据句可追溯与六维归因稳定性作为首版核心价值。
- **简化策略（Simpler）**：
  - 界面语言首版仅支持 **English / 简体中文**。
  - RBAC 首版采用 **3 角色固定模板**（管理员、运营、只读），不提供自定义角色编辑器。

## 二、分析能力需求（模型侧）

分析能力**可配置**：可选择使用**自建/微调模型**，也可选择**对接外部大模型 API**。管理员在「接口配置」维护分析源池（含默认项），业务用户在评论洞察任务发起时选择本次分析源。自建/微调模型基于**直接下载的数据集**进行训练与评估，与线上评论抓取 API 数据分离。以下为**结果级要求**——无论采用哪种方式，系统对外呈现的分析结果均需满足这些要求；选用自建/微调模型时由本系统保证达标，选用外部大模型 API 时需在配置中约定请求/响应规范并确保输出符合要求。具体技术选型与微调方案在后续 spec/plan 阶段确定。

### 2.1 情感分析

| 要求项 | 说明 |
|--------|------|
| **准确率** | 情感分类（如正/中/负）在预留测试集上 **准确率 ≥ 85%**。选用自建/微调模型时为上线与迭代的硬性指标；选用外部大模型 API 时建议约定同等或可接受的质量要求。 |
| **输出** | 每条评论输出情感标签及置信度（或分数），供可视化与对比分析使用。 |
| **实现方式** | 可配置为自建/微调模型（如基于预训练模型微调）或对接外部大模型 API；接口配置中选定分析来源后，系统统一按上述输出规范调用并落库。 |

### 2.2 六维痛点/维度分析

| 要求项 | 说明 |
|--------|------|
| **六个维度** | 必须覆盖：**优点、缺点、退货原因、购买动机、用户期望、应用场景**；每条评论可归属多个维度，并产出维度下的标签/关键词。 |
| **可解释与可归因** | 每个痛点/标签需能**归因到具体评论或片段**（即「谁说的」），以便展示证据句与高亮。 |
| **痛点评分与优先级** | 支持对痛点/标签做**可量化的排序依据**（如综合频次、情绪强度、时效、趋势等），便于「痛点排行榜」与优先级决策。 |
| **实现方式** | 可配置为自建/微调方案（如 **RoBERTa 评论级情感 + 六维词典与可解释匹配规则 + BERTopic 离线主题辅助与人工审核回灌**）或对接外部大模型 API；与情感分析配合形成「情感 + 词表/主题归属 → 归因到六维」的可解释链路，输出格式与六维、证据句要求由配置约定。 |

### 2.3 证据句与可解释性

| 要求项 | 说明 |
|--------|------|
| **证据句** | 每个痛点/标签需关联**证据句**（原始评论或片段），用于前端「证据句高亮」与导出。 |
| **存储** | 证据句与评论 ID、维度、标签一并落库，支持按痛点筛选与展示。 |

### 2.4 新痛点发现与词典回灌（可选，可放在后续迭代）

| 要求项 | 说明 |
|--------|------|
| **目标** | 在短评场景下发现**新增痛点或主题**，并将结果回灌到系统词典/配置，使六维分析可持续迭代。 |
| **范围** | 若首版不实现，需在 spec/plan 中明确为 v2 或后续迭代；若实现，需约定回灌流程与版本管理，避免覆盖错误。 |

### 2.5 与「接口配置」的边界

| 能力 | 数据/能力来源 | 说明 |
|------|----------------|------|
| **评论数据** | 评论抓取 API（必配） | 系统只负责调用配置好的 API 获取评论并落库，不实现爬虫本身。 |
| **情感 + 六维痛点** | 可配置：自建/微调模型 或 外部大模型 API（支持多个） | **主要使用场景：评论洞察**——对抓取到的评论做情感与六维分析；管理员在接口配置维护分析源列表（含默认项），业务用户在评论洞察输入区底部按任务选择分析源；两种方式均需满足 2.1、2.2、2.3 的结果级要求。 |
| **结论卡片（对比分析）** | 内置规则 或 智能 Agent API（可选） | **主要使用场景：对比分析**——基于两个商品已落库的分析结果生成结论文案；可先由规则生成，若配置了智能 Agent API 则可用其增强。 |
| **界面翻译（英文分析展示文本）** | 翻译 API（可选，建议配置） | 将**英文分析展示文本**翻译为**当前界面语言**（仅展示层，不改变分析输入）；当界面语言为 English 时不显示译文；当界面语言非 English 时在英文下方紧接展示译文（上下结构，机器翻译提示）；证据句区域不参与翻译。 |

**对应关系小结**：分析用模型/大模型 API 服务于**评论洞察**与六维/关键词/结论生成；智能 Agent API 服务于**对比分析**中的结论卡片；**翻译 API** 服务于**英文分析展示文本**相对**界面语言**的展示层翻译，与分析与结论生成解耦；证据句区域不翻译。**界面语言**（English / 简体中文）由前端 i18n 与全局切换控制，与翻译 API 的目标语种对齐。分析用 API 与 Agent 在技术实现上可共用同一底层大模型，但与翻译接口的用途、入参、出参均不同。

## 四、功能需求详述

### 4.1 评论洞察

#### 4.1.1 用户侧需求

- 支持输入 **一个或多个商品标识**（`platform + product_id`，如 Amazon ASIN / JD skuId）或商品链接/网址。
- 在输入区（商品标识/网址输入框下方）提供**分析源选择器**，支持本次任务选择：`自建/微调模型` 或 `外部大模型 API`（从已配置列表中选择）。
- 触发后，由系统完成：抓取评论 → 调用模型分析 → 展示分析结果与可视化。
- 用户可查看：痛点排行、维度词云、时间趋势、证据句、筛选与导出；若评论含 SKU 可查看 SKU 维度对比。
- 默认展示**英文分析结果**；当界面语言为 English 时仅展示英文；当界面语言为简体中文或其他语言时，在英文下方紧接展示**译文（机器翻译）**（上下结构）；证据句区域仅展示对应原始评论片段（不翻译）；若未配置翻译 API 且界面语言非 English，则仅展示英文并提示「翻译服务未配置」。

#### 4.1.2 系统侧需求

**输入与抓评**

- 输入：单个或多个商品标识（`platform + product_id`，或可解析出商品标识的商品 URL）；首版按**多站点可扩展**设计（站点可通过配置指定），评论原文随站点语言返回；对非英文站点在分析前统一转换为英文输入。
- 分析源选择：前端提交任务时需携带 `analysis_provider_id`（来自输入区底部选择器）；若用户未手动选择，则回退到接口配置中的默认分析源。
- 系统调用已配置的 **评论抓取 API**，获取该商品下的评论数据（如内容、时间、评分、SKU 等）。
- 抓取结果落库，并与任务/商品维度关联，便于后续分析结果关联与对比。

**情感与痛点分析**（见第二节「分析能力需求」与 4.3：由**自建/微调模型部署**或**分析用模型/大模型 API** 提供；接口配置负责维护分析源池，评论洞察任务在输入区选择本次分析源，非「智能 Agent API」）

- 对抓取到的评论进行 **情感分析**（如正面/中性/负面及置信度），准确率目标 ≥85%。
- 进行 **六维痛点/维度分析**，需明确输出以下 6 个方面：
  1. **优点**：用户认可的产品优点。
  2. **缺点**：用户指出的产品缺点。
  3. **退货原因**：评论中提及的退货或不满原因。
  4. **购买动机**：驱动用户购买的因素。
  5. **用户期望**：用户希望改进或期待的功能/体验。
  6. **应用场景**：用户描述的使用场景。

- 分析结果（情感标签、六维标签/关键词、证据句等）需 **持久化到数据库**，并与评论、商品、任务关联。

**可视化面板**

- **痛点/维度排行榜**：按维度（优点/缺点/退货原因等）或按具体标签排序展示（如频次、强度、趋势综合得分）。
- **维度词云**：基于分析结果的关键词/短语词云，可区分维度或情感。
- **时间趋势**：评论带时间戳，支持按年/月/日展示评论量或六维/情感占比随时间的变化。
- **证据句高亮**：左侧展示当前维度下的痛点列表，用户点击某痛点后，右侧展示与该痛点相关的评论（证据句），评论中与痛点相关的关键词予以高亮；**证据句区域仅展示对应原始评论片段（不翻译）**（布局见 4.5）。
- **SKU 对比**：若评论带 SKU 信息，支持按 SKU 查看情感与痛点分布对比。
- **筛选与导出**：按时间、情感、维度、关键词等筛选；支持导出为 Excel/CSV（列表或汇总）。

#### 4.1.3 业务流程简述

1. 用户在「评论洞察」页输入商品标识（`platform + product_id`，或网址），并在输入区底部选择本次分析源（自建/微调模型或某个外部大模型 API）。
2. 系统校验并调用评论抓取 API，抓取评论并写入数据库。
3. 系统调用**分析用模型/大模型 API**（自建/微调或外部大模型，见接口配置）产出情感与六维结果并落库。
4. 前端从后端拉取分析结果，渲染可视化面板；用户可筛选、查看证据句、导出。

### 4.2 对比分析

#### 4.2.1 用户侧需求

- 用户可 **选择两个商品（`platform + product_id`）** 进行对比。
- 对比所使用的数据来源于 **评论洞察** 已分析过的数据（即先有洞察任务，再选其中两个做对比）。
- 展示形式：情感对比、痛点对比、关键词差异、结论卡片，便于快速理解差异与决策。
- 对比视图中的分析结果相关内容（情感/痛点/关键词/结论卡片）统一以英文展示；当界面语言非 English 时，在英文下方紧接展示译文（上下结构，机器翻译提示）；若对比页支持下钻到证据句，则仅展示对应原始评论片段（不翻译）。

#### 4.2.2 系统侧需求

- **数据来源**：仅对已在「评论洞察」中完成抓取与分析的商品提供对比；若某商品无分析数据，需提示用户先执行评论洞察。
- **情感对比**：两商品的情感分布对比（如正/中/负占比、平均分或强度）。
- **痛点对比**：六维（优点、缺点、退货原因、购买动机、用户期望、应用场景）在两商品上的分布或排名对比。
- **关键词差异**：两商品的高频/特征关键词差异（如仅 A 有、仅 B 有、共有但权重不同）。
- **结论卡片**：系统生成简短结论卡片（可基于规则或可选配置的智能 Agent API），概括两商品在情感、痛点、关键词上的主要差异与建议（如「A 在物流差评更多」「B 更常被提及性价比」等）。

#### 4.2.3 业务流程简述

1. 用户在「对比分析」页从已分析商品列表中选择两个商品。
2. 系统校验两商品均有分析数据，从库中聚合情感、痛点、关键词等结果。
3. 前端展示情感对比、痛点对比、关键词差异、结论卡片；结论卡片可由后端规则或 Agent 生成。

### 4.3 接口配置

#### 4.3.1 用户侧需求

- 用户可在系统内 **配置评论抓取 API**（如 URL、鉴权方式、参数说明）。
- 用户可 **配置分析用模型/大模型 API**（主要用于**评论洞察**：对评论做情感与六维分析），并支持维护**多个分析源**（含自建/微调与多个外部大模型 API）；评论洞察页输入区底部可按任务选择其中一个分析源执行。
- 用户可 **配置智能 Agent API**（可选，主要用于**对比分析**：生成结论卡片；不配则结论由规则生成）。
- 用户可 **配置翻译 API**（可选）：用于两类场景——
  - **展示层**：当界面语言非 English 时，将英文分析展示文本翻译为对应界面语言并按 4.5 以上下结构展示；**证据句不参与翻译**。
  - **分析输入**：当站点原文非英文且分析模型仅在英文上训练时，将站点文本翻译为英文作为模型输入（analysis_input_en）。
  未配置时：若仅涉及展示层翻译，则仅展示英文并提示「翻译服务未配置」；若涉及分析输入翻译，分析质量可能下降并提示「翻译服务未配置」。
- 配置后，各模块按场景读取对应配置并调用外部服务。

#### 4.3.2 系统侧需求

- **评论抓取 API 配置**：至少包含接口地址、请求方法、鉴权信息（如 API Key/Token）、必要参数映射（如 `platform + product_id` 如何传入）；支持测试连接或试抓一条，以验证配置有效。
- **分析用模型/大模型 API 配置**：支持维护分析源列表（`provider_id`、来源类型=自建/外部、显示名称、接口地址、鉴权、请求/响应格式、是否启用、是否默认）；**仅用于评论洞察**；无论后端实际为自建模型服务还是外部大模型，均通过统一配置结构调用。
- **运行时选择规则**：评论洞察任务优先使用页面提交的 `analysis_provider_id`；未提交时使用默认分析源；若所选分析源被禁用/不可用，需给出明确错误并引导切换其他分析源。
- **智能 Agent API 配置**：至少包含接口地址、鉴权、请求/响应格式说明（或预设模板），便于将**两商品对比摘要**传给 Agent 并解析结论文案；主要用于对比分析；可选支持简单测试调用。
- **翻译 API 配置**：接口地址、鉴权、请求/响应格式；支持用于两类翻译：
  - **展示层**：源语言为英文分析展示文本；目标语言与用户界面语言联动（当界面语言非 English 时按上下结构展示英文+译文规则）；证据句不参与翻译。
  - **分析输入**：当站点原文非英文且模型仅在英文训练时，将源文本翻译为英文作为模型输入。
  - 用于展示层翻译与分析输入翻译均**不替代**分析用模型的结构化输出；支持简单测试调用（如试译一句）。
- 配置信息需 **安全存储**（如加密或仅后端可读），且按环境或租户隔离（若多租户）。
- 仅 **具备相应权限的角色**（如管理员）可查看与修改接口配置。

### 4.5 界面语言、站点原文与内容翻译（展示层）

**目标**：第一次做产品时，将三件事分开，避免混淆：**① 页面用什么语言**、**② 业务数据按站点显示什么语言**、**③ 要不要把业务内容译成页面语言**。

#### 4.5.1 界面语言（全站 UI）

- 用户在显著位置（如顶栏）可选择 **English** 或 **简体中文**。
- 作用范围：**系统固定文案**——菜单、按钮、表单标签、提示语、空状态、图表标题等（通过前端 i18n 实现）。
- 与「评论正文语言」无关：切换界面语言**不**改变评论原文（站点语言文本）本身。

#### 4.5.2 站点原文（用于证据句，不翻译）

- **首版**按多站点架构设计，可接入不同国家/语种站点（具体首批站点清单在 spec 中确认）。
- **证据句**展示：证据句对应到原始评论片段，**始终展示站点原文**（原始语言文本）。
- **证据句不翻译**：证据句区域不展示译文，即使页面语言为非英文，也仅展示站点原文，避免影响可信度与可追溯性。

#### 4.5.3 内容翻译（分析展示英文 + UI 语言译文，下方紧接）

- **分析与可视化展示语言**：所有与“情感/六维痛点/关键词/痛点排行/结论卡片”等分析相关的文本，统一以 **英文**展示（即分析结果英文化展示）。
- **译文触发条件**：当页面语言（UI语言）为 **English** 时，仅展示英文；当页面语言为简体中文或其他语言时，在英文下方紧接展示译文（机器翻译提示），形成**上下结构：英文在上、译文在下**。
- **翻译对象**：仅对“英文分析展示文本”翻译（如标签/关键词/痛点名称/词云词/对比页关键词/结论卡片等）。**证据句不参与翻译**。
- **缓存**（建议）：可对英文分析文本按原文哈希 + 目标 UI 语言 + 站点标识缓存译文，减少重复计费与降低延迟。
- **版式**：英文原文完整展示；译文紧接在英文下方，并与英文区分提示（如「译文（机器翻译）」）。

#### 4.5.4 未配置翻译 API

- 当 UI 语言为 English：仅展示英文，不涉及翻译。
- 当 UI 语言非 English 且未配置翻译 API：仍展示英文分析结果；在需要译文的区域或提示处说明「翻译服务未配置」，不阻断主要分析结果展示。

#### 4.5.5 六维等固定维度名

- PRD 中六维名称以中文书写为业务定义；在 **English** 界面下，维度标签应对应展示英文文案（如 Pros / Cons / …），在 spec 或 i18n 词表中维护，与 4.5.1 一致。

### 4.6 痛点审核与词典管理（离线增强）

#### 4.6.1 用户侧需求

- 管理员可在「痛点审核」查看 BERTopic 离线发现的候选痛点主题（关键词、代表证据句、频次、质量分）。

- 权限约束：仅管理员可审核、发布、回滚；运营与只读角色仅可查看（若授权）。

## 五、系统流程（端到端）

### 5.1 评论洞察端到端流程

```
用户输入商品标识（platform + product_id）/商品网址
    → 在输入区底部选择本次分析源（自建/微调模型或某个外部大模型 API）
    → 系统解析平台与商品标识，调用评论抓取 API
    → 评论数据落库（评论表 + 与商品/任务关联）
    → 若站点原文语言非英文且模型仅英文训练：将站点文本翻译为英文作为分析输入（analysis_input_en）
    → 按所选 analysis_provider_id 调用对应分析服务（基于英文分析输入执行情感与六维痛点；**非**智能 Agent API）
    → 分析结果落库（情感、六维标签、证据句等）
    → 前端拉取数据，展示可视化面板：
        痛点排行榜、词云、时间趋势、证据句高亮、SKU 对比、筛选与导出
    → （若界面语言非 English 且已配置翻译 API）按 4.5 规则展示英文+译文（英文上、译文下）；若为 English 则仅展示英文
```

### 5.2 对比分析端到端流程

```
用户选择两个已分析商品
    → 系统校验数据存在，从库中聚合两商品的分析结果
    → 计算/生成：情感对比、痛点对比、关键词差异、结论卡片
    → 前端展示对比视图与结论卡片
    → （若界面语言非 English 且已配置翻译 API）按 4.5 规则展示英文+译文（英文上、译文下）；若为 English 则仅展示英文
```

### 5.3 任务中心、接口配置、系统设置与词典治理

- **任务中心**：统一展示评论洞察任务与离线主题发现任务状态，支持筛选、失败原因查看与重试。
- **接口配置**：管理员在「接口配置」页填写/修改评论抓取 API、分析用模型/大模型 API（评论洞察用，支持多分析源与默认项）、智能 Agent API（对比分析结论用，可选）、翻译 API（分析输入翻译 + 英文分析展示文本译至界面语言，可选）并保存；各模块在运行时按场景读取对应配置发起调用。
- **系统设置**：管理员在「系统设置」中进行用户与权限管理；登录与权限校验贯穿所有需要鉴权的接口与页面。
- **痛点审核与词典管理**：离线任务生成候选痛点后，由管理员审核（通过/驳回/暂缓）；通过项进入词典新版本发布流程，发布后生效；支持版本回滚与审计追踪。

## 六、非功能需求

### 6.1 性能与容量

- **评论洞察任务 SLA**：在单任务评论量 1000 条场景下，端到端总完成时长 **P95 <= 180s**，任务成功率 **>= 99%**。
- **对比分析接口 SLA**：基于已落库数据聚合查询，接口响应 **P95 <= 2s，P99 <= 3s**。
- **正常负载口径**：支持并发评论洞察任务 **5** 个；实时查询吞吐 **QPS 20**。

### 6.2 安全与合规

- 接口配置中的密钥等敏感信息加密存储、不在前端展示完整密钥。
- 用户与权限数据受保护；操作日志需记录关键操作（如配置变更、批量分析触发、角色变更）。
- 本项目为个人项目，当前无强制外部法规与企业审计条款；若后续接入企业场景，再补充数据保留与删除策略。

### 6.3 可用性与体验

- 评论洞察进行中应有明确进度或状态提示（如抓取中、分析中、完成）。
- 对比分析若某商品无数据，应明确提示并引导用户先去评论洞察。
- 业务内容区默认仅展示**英文分析结果**；当界面语言非 English 时会自动展示译文（上下结构、机器翻译提示）；证据句区域仅展示对应原始评论片段（不翻译）；若未配置翻译 API 且界面语言非 English，则仅展示英文并提示「翻译服务未配置」，保证分析结果可用性与可信度。
- 系统月度可用性目标 **>= 99.5%**。

### 6.4 兼容与扩展

- 评论抓取 API、分析用模型/大模型 API、智能 Agent API、翻译 API 通过配置接入，便于更换或扩展数据源与模型，无需改代码（仅改配置或适配器）。

| 词典管理 | 当前生效版本、词条列表、来源与状态、版本发布与回滚、变更日志 | 仅管理员可发布/回滚 |

## 九、验收标准（摘要）

- **分析能力**：情感分析在预留测试集上准确率 ≥ 85%；六维痛点分析可产出维度与标签，并关联证据句；痛点评分支持排序与下钻。
- **评论洞察**：输入平台商品标识（`platform + product_id`）可完成抓评、分析、落库，并在面板中正确展示六维痛点、维度词云、时间趋势、证据句及导出；SKU 对比在评论数据含 SKU 时可用；在 1000 条评论场景下满足端到端 **P95 <= 180s** 与成功率 **>= 99%**。
- **对比分析**：选择两个已分析商品可展示情感对比、痛点对比、关键词差异及结论卡片。
- **任务中心**：可统一查看评论洞察与离线任务状态；失败任务可查看错误并触发重试；权限控制符合角色边界。
- **接口配置**：可保存并生效评论抓取 API、分析用模型/大模型 API（支持多分析源 + 默认项）、智能 Agent API（可选）、翻译 API（可选）配置；评论洞察可按任务选择分析源并正常抓取与分析；对比分析在配置 Agent 时能生成增强结论。
- **界面语言与自动对照**：全站可切换 English / 简体中文，系统文案正确切换；分析与展示文本统一以英文展示；当界面语言非 English 且已配置翻译 API 时，显示英文 + 译文（英文上、译文下紧接展开，机器翻译提示）；界面语言为 English 时不显示译文；未配置且界面语言非 English 时仅展示英文并提示「翻译服务未配置」；证据句区域不翻译；行为符合 4.5.3。
- **系统设置**：可管理用户与权限，且前端菜单与接口按权限正确展示与拦截；RBAC 满足固定 3 角色模板（管理员、运营、只读）与对应权限边界。
- **痛点审核与词典管理**：BERTopic 候选项可审核并形成发布词典版本；发布后生效且支持回滚；审核/发布/回滚操作可追溯。

### 11.1 名词说明

| 术语 | 说明 |
|------|------|
| platform + product_id | 跨平台统一商品标识。`platform` 表示平台（如 amazon/jd/pdd），`product_id` 表示该平台的商品唯一标识（如 ASIN、skuId、goods_id）。 |
| 六维分析 | 优点、缺点、退货原因、购买动机、用户期望、应用场景。 |
| 证据句 | 支撑某痛点/标签的原始评论或评论片段。 |
| 痛点评分 | 对痛点/标签的可量化排序依据，可综合频次、情绪强度、时效、趋势等。 |
| 自建/微调模型 | 经自训练或微调得到的情感/痛点分析模型，可部署在本系统后端并由「分析用模型/大模型 API」配置指向该服务；与对接**外部大模型 API** 做分析为两种并列选项，由接口配置择一。 |
| 评论抓取 API | 本系统调用的、用于按 `platform + product_id` 获取评论数据的外部接口（必配）。 |

| 翻译 API | 本系统可选的、用于两类场景的外部或自建接口：用于分析输入翻译（当站点非英文且模型仅英文训练时，生成 `analysis_input_en`）；用于展示层翻译（当界面语言非 English 时，将英文分析展示文本翻译为当前界面语言并以上下结构展示；证据句区域不翻译）。目标语种随界面语言联动。 |

### 11.2 评论抓取 API 能力说明（计划采用）

本产品计划采用的评论抓取 API 具备以下能力，用于与「接口配置」及数据落库字段对齐：

- **数据内容**：从 Amazon **实时**获取评论，包括但不限于——**评分（ratings）、标题（titles）、正文/描述（descriptions）、用户反应（user reactions）、评论日期（review dates）、图片（images）** 等，满足情感与六维痛点分析及证据句展示所需。
- **输出格式**：支持 JSON、CSV、Excel、XML 或 HTML 等格式，便于系统解析、落库与后续分析。
- **能力定位**：快速、准确，适用于对评论进行深度洞察与结构化分析；系统通过配置该 API 的地址与鉴权信息，按 `platform + product_id` 请求评论并写入数据库，不自行实现爬虫逻辑。

（具体 API 提供商、请求参数与响应结构在技术设计阶段与接口配置中约定。）
