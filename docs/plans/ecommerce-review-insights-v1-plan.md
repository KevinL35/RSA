## Two-Stage Delivery

### Stage A - 模型微调与评估（先行阶段）

- 明确情感与六维任务定义、标签规范、评估口径。
- 构建训练/验证/测试数据集（与线上抓评数据链路分离）。
- 完成分析链路：**RoBERTa 总体情感** → **六维词典 + 可解释匹配规则归因（证据句与高亮 span）** → **BERTopic 离线新痛点发现与人工审核回灌词典**。
- 完成模型微调、离线评估、推理服务封装（可被 `analysis_provider_id` 调用）。
- 产出模型版本与质量基线，为 Stage B 提供稳定分析源。

### Stage B - 前后端产品化与上线（后续阶段）

- 基于 Stage A 的分析服务接入产品链路。
- 完成评论洞察、对比分析、翻译展示、固定模板 RBAC。
- 完成任务中心（任务状态、错误定位、重试能力）。
- 完成痛点审核与词典管理菜单（管理员审核、发布、回滚）。
- 完成性能压测、可观测性与上线验收。

## Architecture Plan

### 1) 端到端数据链路

- **采集层**：接收 `platform + product_id` 或 URL，调用评论抓取 API，持久化原始评论。
- **分析层**：根据 `analysis_provider_id` 调用分析源，按「RoBERTa 情感 → 六维词典/规则归因 + 证据句」在线链路产出情感+六维+证据句（主题向量可分表存储，供离线对齐与重算）。
- **发现层（离线）**：使用 BERTopic 从评论语料挖掘新痛点主题，产出候选词并回灌词典版本。
- **展示层**：提供洞察页与对比页查询接口，支持筛选、下钻、聚合。
- **翻译层**：仅翻译分析展示文本；证据句保持原文。
- **权限层**：固定 3 角色（管理员/运营/只读）统一控制前后端访问。

### 2) 关键设计决策

- **分析源策略**：任务优先使用用户选择分析源，未选择回退默认分析源。
- **情感主模型**：使用 RoBERTa 承担总体情感分类（正/中/负 + 置信度），作为上游稳定信号。
- **六维归因策略**：以**版本化六维词典**（关键词/短语）+ **可解释匹配规则**（长词优先、多命中排序、冲突处理）将评论关联到六维痛点；证据句为**原文片段**，高亮为**词典命中 span**；BERTopic 仅参与**离线**候选发现与回灌，不单独作为在线「切面分类器」。
- **新痛点发现策略**：BERTopic 仅做离线发现，不直接覆盖在线分类；回灌词典采用“候选 -> 人工复核 -> 发布”流程。
- **翻译策略**：UI 为 English 时仅英文；非 English 展示英文+译文上下结构；翻译不可用时不阻断主流程。
- **RBAC 策略**：先固定模板角色，避免首版引入复杂权限编排。
- **性能策略**：洞察任务异步化（任务队列或后台任务），对比查询走聚合缓存/索引优化。

### 3) 数据模型规划（逻辑级）

- `insight_tasks`：任务元数据（状态、开始/结束时间、分析源、错误信息）。
- `offline_tasks`：离线任务元数据（任务类型、窗口范围、状态、批次号、错误信息）。
- `reviews`：原始评论数据（平台、商品、内容、时间、SKU 等）。
- `review_analysis`：情感标签、六维标签、关键词、证据句映射。
- `taxonomy_dictionary_entries`：词条、六维归属、来源、状态、权重、生效版本。
- `taxonomy_dictionary_versions`：六维词典版本、词条、来源、发布时间与回滚信息。
- `topic_discovery_runs`：BERTopic 运行批次、主题质量指标、候选词与审核结果。
- `topic_review_decisions`：候选审核动作（通过/驳回/暂缓）、审核人、时间、备注。
- `compare_snapshots`（可选）：对比聚合缓存，降低重复计算。
- `api_configs`：抓取/分析/翻译/Agent 配置（敏感字段加密）。
- `users`, `roles`, `user_roles`：固定模板 RBAC 与用户关系。

### 4) Stage A 实施口径（HuggingFace）

- **实现语言**：Python（训练与离线发现主语言），服务接入层按现有后端技术栈封装。
- **核心工具链**：`transformers`、`datasets`、`evaluate`、`scikit-learn`、`bertopic`、`sentence-transformers`、`pandas`。
- **模型与规则职责分工**：
  - RoBERTa：总体情感分类（positive/neutral/negative + confidence）。
  - 六维归因引擎：基于当前生效 `taxonomy` 版本对 `analysis_input_en` 做匹配，输出维度、痛点标签、证据片段、`highlight_spans`；规则文档与版本一并发布。
  - BERTopic：离线主题发现，输出「候选新痛点词/主题关键词」，经人工审核后写入词典新版本。

### 5) 训练数据契约与清洗要求

- **统一字段（最小集）**：`id`, `platform`, `product_id`, `raw_text`, `analysis_input_en`, `lang`, `created_at`, `source`, `split`。
- **RoBERTa 数据集字段**：`text_en`, `label_sentiment`。
- **归因/证据抽检集字段（可选，用于规则与词典迭代）**：`text_en`, `dimension_6way`, `pain_label`, `evidence_quote`（原文截断）, `expected_highlight_tokens`（词表级期望，非模型训练硬依赖）。
- **BERTopic 语料字段**：`text_en`, `platform`, `product_id`, `created_at`（无标注即可）。
- **清洗必做项**：去重、去噪（URL/HTML/模板垃圾文本）、长度过滤、语言识别、标签一致性抽检。
- **语言策略**：分析输入统一英文（`analysis_input_en`），保留原文 `raw_text` 供证据句展示与追溯。

### 6) 评估口径与发布契约

- **RoBERTa 评估**：Accuracy、Macro-F1、Confusion Matrix；上线门槛 Accuracy >= 85%。
- **归因与证据评估**：词典命中率（抽样）、多命中冲突解决一致性、证据句可追溯到 `review_id` 的比例、人工抽检通过了然度（可按导师要求定量化表格）。
- **BERTopic 评估**：主题连贯性、主题覆盖率、候选词人工通过率。
- **发布产物**：模型权重、tokenizer、评估报告、**归因规则/词典版本**变更记录、推理与归因接口契约。
- **版本规范**：建议采用 `sentiment-vX.Y.Z`、`taxonomy-vX.Y.Z`、`topic-pipeline-vX.Y.Z`（离线批次与脚本配置）分别版本化。

## 工程里程碑（M0–M3）

### M0 - 模型微调可用（Stage A）

- 完成数据规范与标注口径冻结（情感 + 六维）。
- 完成 RoBERTa 情感模型微调与离线评估。
- 完成六维词典首版、归因匹配规则与冲突策略，并完成证据句与高亮的抽检评估。
- 完成 BERTopic 离线主题发现流程与词典回灌机制（含人工复核）。
- 完成模型服务化与接口契约冻结（统一输入输出）。
- 完成 HuggingFace 训练脚本与可复现配置（含随机种子、数据版本、超参记录）。

**退出条件**
- 情感分类离线准确率达到 PRD 目标（>=85%）。
- 六维输出可稳定产出标签/关键词/证据句映射。
- BERTopic 可稳定产出候选新痛点，且至少完成一轮“发现 -> 复核 -> 词典发布”闭环演练。
- 推理服务可通过统一 `analysis_provider` 协议接入。
- 训练与评估流程可复现（同一数据版本下关键指标波动在可接受区间）。

### M1 - 基础链路可用（Stage B / P0 核心）

- 完成评论洞察最小闭环：输入、抓评、分析、落库、查询。
- 完成多分析源任务级选择与默认回退。
- 完成洞察页基础可视化查询接口。
- 完成任务中心最小功能（任务列表 + 状态 + 失败原因 + 重试）。

**退出条件**
- 可创建并完成洞察任务。
- 可返回情感/六维/证据句结构化结果。
- 可在任务中心查询任务状态，并对失败任务执行可控重试。

### M2 - 对比、翻译、权限联动（Stage B）

- 完成对比分析接口（情感/痛点/关键词/结论卡片）。
- 完成双语 UI 与翻译展示规则。
- 完成固定模板 RBAC（前后端一致拦截）。
- 完成痛点审核页与词典管理页（版本发布/回滚）。

**退出条件**
- 两个已分析商品可完成对比查询。
- 非 English 下展示英文+译文，翻译缺失时有提示不阻断。
- 3 角色访问边界满足 spec 验收。
- 管理员可完成候选审核与词典版本发布，且支持回滚。

### M3 - 稳定性与上线准备（Stage B）

- 性能压测与瓶颈优化。
- 指标与监控看板补齐（SLA、成功率、可用性）。
- 回归测试与上线清单收敛。

**退出条件**
- 达成评论洞察和对比接口 SLA。
- 月可用性目标具备可观测性（>=99.5%）。

## 毕业论文专项

### 研究目标与可写点

| 维度 | 内容 |
|------|------|
| **问题** | 短文本评论下痛点发现难解释、难量化；应用侧多停留在情感或关键词罗列，缺排序、证据与评估口径。 |
| **目标** | 构建 B/S 系统：评论级情感（准确率 ≥85%）、六维痛点挖掘与量化评分、证据句原文展示与高亮、批量处理与可视化面板。 |
| **创新/工作量落点** | ① 适配电商短文本的主题/BERTopic 与业务词典耦合；② 融合频次、情感强度、时效、趋势的痛点评分；③ 端到端工程闭环（抓取适配—分析—落库—面板—词典版本）。 |
| **明确不做（边界）** | 不自研爬虫；不单独将「方面级序列标注」子模型作为毕设必交付物；标注/训练平台产品化非必做。 |

### 答辩用技术路线（简图）

与上文「端到端数据链路」一致，以下便于论文「总体设计」直接引用：

```
评论抓取 API → 原始评论落库（raw_text）
       ↓
预处理：清洗 / 去重 / 语言识别 → 统一分析输入 analysis_input_en（与 spec 一致）
       ↓
RoBERTa：整条评论情感 + 置信度（核心可评指标，门槛 ≥85% Acc）
       ↓
痛点绑定：
  · 六维词典词条匹配（关键词/短语，长词优先，版本化）
  · BERTopic 离线主题发现 → 主题与「候选痛点/维度」人工对齐 → 回灌词典
  · 可选：主题概率 + 词表命中 + 情感极性/强度 的加权打分
       ↓
痛点评分模块：频次、负面情绪强度（来自 RoBERTa）、时间衰减、环比/趋势（按周或批次）
       ↓
FastAPI：任务编排、批处理、模型调用、写回 review_analysis / 聚合表
Supabase：任务状态、RBAC、配置、词典版本、审核日志
Vue：洞察页、对比页、任务中心、（可选）审核与词典管理页
```

**方法论说明**：证据高亮来自**词典命中 span**与**主题归属**（离线对齐后反映在词表中）；多要点长句的精度主要依赖**词表维护**与**BERTopic 候选审核**。

## Risk & Mitigation（工程）

- **模型质量风险**：离线指标达标但线上泛化不足。  
  缓解：预留跨站点验证集，发布前进行抽样人工复核。
- **标签漂移风险**：不同批次数据导致六维标签分布偏移。  
  缓解：建立标签分布监控与回归评估基线。
- **词典/规则归因风险**：词表缺失、多命中冲突或新说法未覆盖导致归因漂移。  
  缓解：版本化词典、明确冲突处理优先级、BERTopic 候选回灌与人工审核、定期抽检与回归集。
- **BERTopic 噪声主题风险**：离线主题质量波动导致错误回灌。  
  缓解：设置主题质量阈值（连贯性/覆盖率），未达阈值不进入候选池。
- **外部 API 稳定性风险**：抓评/分析/翻译接口波动。  
  缓解：超时重试、熔断降级、错误码标准化。
- **分析结果一致性风险**：不同分析源结果分布差异大。  
  缓解：统一响应契约、最小质量校验、提供分析源标识透传。
- **性能风险**：洞察任务高峰导致堆积。  
  缓解：任务并发控制、分批处理、热点聚合缓存。
- **权限风险**：前后端策略不一致导致越权。  
  缓解：后端作为唯一裁决点，接口鉴权测试覆盖。

## Dependencies

- 评论抓取 API 可用且字段契约稳定。
- Stage A 可用的分析源（自建/微调优先）与版本化发布机制。
- 翻译 API 为可选依赖，但需有“未配置降级路径”。
- 认证系统与用户基础表可用（如 Supabase Auth）。

## Validation Plan

- 模型验证：RoBERTa 准确率；六维归因规则与词典质量（抽检/一致性）；证据句可追溯；BERTopic 主题质量 + 小流量线上抽检。
- 功能验证：按 spec 的 P0/P1 接受标准逐项验收。
- 性能验证：1000 条任务压测、对比接口压测、并发/QPS 压测。
- 权限验证：3 角色正反用例 + 关键越权用例。
- 稳定性验证：故障注入（API 超时/错误）与降级行为验证。
