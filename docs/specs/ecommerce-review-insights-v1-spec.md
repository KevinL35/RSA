# 电商评论情感与痛点分析系统 - Feature Spec (v1)

## Problem Statement

电商运营与产品团队在分析商品评论时，常常无法快速、稳定地定位真实痛点并追溯证据，导致选品与优化决策依赖经验而非结构化数据。当前流程缺少统一的评论洞察、对比分析和权限治理能力，跨角色协作效率低。若不解决，该项目难以在有限时间内形成可重复复用的决策闭环，且问题定位准确度无法稳定提升。

## Goals

- 在 MVP 范围内上线评论洞察、对比分析、多分析源任务级选择、翻译展示、完整 RBAC，形成可用闭环。
- 提升问题定位准确度（核心 Why）：六维痛点均可追溯到证据句，支持稳定归因与下钻分析。
- 满足性能与可用性目标：评论洞察任务（1000 条）P95 <= 180s 且成功率 >= 99%；对比分析接口 P95 <= 2s、P99 <= 3s；月可用性 >= 99.5%。
- 在正常负载下稳定运行：并发评论洞察任务 5 个、实时查询 QPS 20。
- 保障角色权限边界清晰：管理员、运营、只读角色在前后端一致生效。

## Analysis Pipeline（自建/微调主线约定）

- **情感**：RoBERTa（或等价微调模型）产出评论级情感极性（如正/中/负）与置信度。
- **六维归因与证据**：依赖**版本化六维词典**与**可解释匹配规则**（含冲突处理）；证据句为原始评论片段，高亮为词典命中 span。BERTopic 仅用于**离线**主题发现；候选词经**人工审核**后回灌词典，再参与在线归因。
- **非目标（v1）**：不要求单独交付「方面级序列标注」子模型；外部大模型作为分析源时，响应需可映射为与本节一致的结构化结果（或由适配层规整）。

## Non-Goals

- 不实现评论抓取爬虫、反爬、抓取调度系统（依赖外部评论抓取 API）。
- 不建设标注平台、训练平台或实验平台产品化能力（仅使用已训练/已配置模型服务）。
- 不在首版支持超过 English/简体中文的 UI 语言（避免分散交付焦点）。
- 不在首版提供 RBAC 自定义角色编辑器（采用 3 角色固定模板，降低复杂度）。
- 不在首版纳入新痛点发现与词典回灌在线流程（保留为后续迭代）。

## User Stories

### 电商运营 / 选品人员

- 作为运营人员，我希望输入商品标识或链接后自动完成抓评与六维分析，以便快速定位商品核心痛点与优势。
- 作为选品人员，我希望对两个已分析商品进行差异化对比，以便更快做出选品与卖点决策。
- 作为运营人员，我希望查看每个痛点对应的证据句原文，以便验证结论可信度与可解释性。
- 作为运营人员，我希望在界面非 English 时看到英文分析与译文上下对照，以便阅读效率更高。

### 产品 / 客服负责人

- 作为产品负责人，我希望按时间趋势观察痛点变化，以便评估改版效果和问题演进。
- 作为客服负责人，我希望从退货原因和用户期望维度下钻证据句，以便优化客服话术与售后策略。

### 系统管理员

- 作为管理员，我希望管理评论抓取、分析源、翻译与 Agent 配置，以便保障业务任务可执行。
- 作为管理员，我希望通过固定角色模板管理系统访问权限，以便降低误配风险并保证审计可读性。

## Requirements

### Must-Have (P0)

#### P0-1 评论洞察端到端流程
- 支持输入 `platform + product_id` 或可解析 URL。
- 支持按任务选择 `analysis_provider_id`，未选择时回退默认分析源。
- 后端完成抓评、分析、落库、可视化查询链路。
- 接受标准：
  - [ ] 输入合法商品标识可触发任务并可查询任务状态。
  - [ ] 分析结果包含情感标签、六维标签/关键词、证据句关联。
  - [ ] 证据句可按痛点下钻展示，且显示原始评论片段。

#### P0-2 对比分析
- 支持从已分析商品中选择两个商品对比。
- 展示情感分布差异、六维痛点差异、关键词差异、结论卡片。
- 接受标准：
  - [ ] 两商品均有分析数据时返回完整对比结果。
  - [ ] 任一商品缺失分析数据时返回明确引导（先完成评论洞察）。
  - [ ] 对比接口满足 P95/P99 SLA 目标。

#### P0-3 翻译展示与双语 UI
- UI 固定文案支持 English/简体中文切换。
- 分析展示文本默认英文；UI 非 English 时展示英文+译文上下结构。
- 证据句区域始终展示原始评论，不翻译。
- 接受标准：
  - [ ] UI 语言切换后菜单/按钮/提示语正确切换。
  - [ ] 非 English 下分析文案显示英文与译文，且有机器翻译提示。
  - [ ] 未配置翻译 API 时不阻断分析结果，仅提示翻译未配置。

#### P0-4 固定模板 RBAC
- 提供固定 3 角色：管理员、运营、只读。
- 前端菜单与后端接口权限保持一致。
- 接受标准：
  - [ ] 管理员可访问接口配置与系统设置。
  - [ ] 运营可访问评论洞察与对比分析，不能修改系统级配置。
  - [ ] 只读只能查看授权页面，无法触发配置变更。

#### P0-5 性能、容量与可用性
- 评论洞察任务：1000 条评论，P95 <= 180s，成功率 >= 99%。
- 正常负载：并发任务 5、查询 QPS 20。
- 月可用性 >= 99.5%。
- 接受标准：
  - [ ] 压测报告覆盖任务链路与查询链路，并符合目标阈值。
  - [ ] 上线后监控看板包含 SLA、成功率、可用性核心指标。

#### P0-6 任务中心
- 提供统一任务中心页面，覆盖评论洞察任务与离线任务状态管理。
- 接受标准：
  - [ ] 任务列表支持按类型、状态、时间筛选。
  - [ ] 支持查看失败原因（错误码/错误描述/失败阶段）。
  - [ ] 对可重试任务提供重试入口，并具备幂等保护。
  - [ ] 角色权限生效：管理员全量可见并可重试，运营按授权可见，只读仅查看。

### Nice-to-Have (P1)

#### P1-1 SKU 维度分布与对比
- 评论含 SKU 时，支持按 SKU 观察情感/痛点分布差异。
- 接受标准：
  - [ ] 存在 SKU 数据时展示 SKU 过滤与分布图。
  - [ ] 无 SKU 数据时页面给出清晰空态提示。

#### P1-2 导出增强
- 支持按筛选结果导出 CSV/Excel（明细与汇总）。
- 接受标准：
  - [ ] 导出文件字段与界面筛选口径一致。
  - [ ] 超过阈值时采用异步导出并可查询状态。

#### P1-3 痛点审核菜单
- 提供管理员审核 BERTopic 候选痛点的页面与流程（通过/驳回/暂缓）。
- 接受标准：
  - [ ] 候选项展示 `topic_id`、关键词、代表证据句、频次、质量分。
  - [ ] 审核动作带操作人、时间与备注，形成可追溯记录。

#### P1-4 词典管理菜单
- 提供词典版本浏览、发布与回滚能力，确保回灌可控。
- 接受标准：
  - [ ] 可查看当前生效版本与词条来源（manual/bertopic）。
  - [ ] 支持发布新版本并回滚到上一个稳定版本。
  - [ ] 仅管理员可发布/回滚，非管理员只读（按授权）。

### Future Considerations (P2)

#### P2-1 新痛点发现与词典回灌
- 增加离线发现新主题与词典版本管理流程。

#### P2-2 多语言扩展
- 在双语稳定后扩展更多 UI 语言与翻译策略。

#### P2-3 RBAC 自定义角色
- 支持自定义角色权限编排、模板继承与变更审批。

## Success Metrics

### Leading Indicators (1-4 周)

- 评论洞察任务成功率 >= 99%（按日/周统计）。
- 评论洞察任务时长（1000 条）P95 <= 180s。
- 对比分析接口时延 P95 <= 2s、P99 <= 3s。
- 任务中心中失败任务可定位率 >= 95%（失败任务存在可读错误信息）。
- 证据句可下钻命中率 >= 99%（痛点项可找到对应证据句）。
- 关键角色权限错误率 < 0.5%（越权或误拦截请求占比）。

### Lagging Indicators (1-3 个月)

- 运营侧“痛点定位效率”提升（从反馈到确认证据句的平均时长下降 >= 30%）。
- 对比分析使用渗透率 >= 60%（有洞察任务的商品中，被用于双商品对比的占比）。
- 关键页面稳定性：月可用性 >= 99.5%。

## Open Questions

- **[Engineering][Blocking]** 痛点评分公式的默认权重（频次/情绪强度/时效/趋势）是否需要配置化？
- **[Data][Blocking]** 六维标签与关键词的标准化词表是否在首版提供最小可用版本？
- **[Engineering][Non-blocking]** 对比分析结论卡片默认采用规则模板时，是否保留可解释字段（来源指标明细）给前端下钻？
- **[Stakeholder][Non-blocking]** 上线后首批支持站点清单（Amazon/JD/其他）的优先级排序。
- **[Stakeholder][Non-blocking]** 痛点审核与词典管理是否作为 v1.1 默认开启，还是先灰度给管理员。

## Timeline Considerations

- 当前无外部强制合规或合同截止日，可按工程节奏推进。
- 建议采用三阶段交付：
  - **Phase 1（基础链路）**：评论洞察 + 多分析源选择 + 基础可视化 + 双语框架。
  - **Phase 2（对比与权限）**：对比分析 + 固定模板 RBAC + 翻译展示联动。
  - **Phase 3（稳定性验收）**：性能压测、SLA 校验、可用性监控与上线准备。
- 依赖项：评论抓取 API、分析源 API、翻译 API（可选）、认证体系（如 Supabase Auth）配置可用。
